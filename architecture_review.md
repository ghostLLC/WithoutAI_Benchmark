1. **评分管线（Assessment Pipeline）设计精良**：6 步流水线职责分离清晰，每步可独立测试

## 二、MBTI 模式改造：可行性分析与完整方案

### 2.1 你的想法理解

将当前 **4 个独立场景各自独立答题** 的模式，改为：
1. 一套统一的测评问卷
2. 前期有一道"场景分流题"
3. 用户选择后，后续只出现与该场景相关的题目
4. 类似 MBTI 那种：所有人进同一个入口，但分流出不同结果

### 2.2 可行性判断：✅ 完全可行

**理由：**

1. **底层模型天然支持**：五维能力模型是跨场景共用的，场景差异仅体现在 `SCENE_WEIGHTS`（维度权重矩阵）上，合并后不影响评分逻辑核心
2. **评分管线无需大改**：`RiskScoreCalculator` 已经按 `sceneId` 选择权重矩阵，只要前端传正确的 `sceneId` 就能工作
3. **现有题库已经具备跨场景通用题**：5 类题目类型（起始方式、关键过程承担、AI 介入位置、脱离 AI 可完成度、依赖惯性）在所有场景中结构一致
4. **产品边界不冲突**：合并场景不等于做"跨场景综合测评"——用户仍然只拿到 **单场景结论**，只是入口体验更统一

### 2.3 与 MBTI 的类比分析

| 维度 | MBTI | WithoutAI（改造后） |
|---|---|---|
| **入口** | 统一入口，不分人群 | 统一入口，不预选场景 |
| **分流机制** | 无显式分流，所有人答完全相同的题 | **前期 1-2 道题显式分流**（用户选择场景） |
| **通用题** | 全部题目通用 | **前段 ~5 题通用**（跨场景共性能力检测） |
| **差异题** | 无 | **后段 ~5 题场景专属**（根据分流结果出现） |
| **维度** | 4 维度 (E/I, S/N, T/F, J/P) | 5 维度 + 场景权重矩阵 |
| **结果** | 16 种人格类型 | 3 档结论 × 4 场景 × 5 风险模式 |

> [!IMPORTANT]
> **关键区别**：MBTI 没有显式分流题——所有人答同一套题。但你的产品 **必须保留场景分流**，因为场景是判断的核心锚点。去掉场景就违反了产品第一原则。
> 
> 更准确的类比是：**MBTI 的入口统一性** + **场景化的内容差异性**。

### 2.4 推荐改造方案

#### 阶段一：统一入口 + 通用题 + 分流题 + 场景题

```
┌─────────────────────────────────────────────────────────┐
│                    统一测评入口                           │
│     "在几分钟内判断你当前使用 AI 的真实边界"              │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│          Part 1: 通用题（3-4 题，所有人都答）              │
│                                                         │
│  检测跨场景共性能力：                                     │
│  · 整体 AI 使用习惯（起始方式）                           │
│  · 脱离 AI 的整体自信度（依赖惯性）                       │
│  · AI 在你日常任务中的角色（AI 介入位置）                  │
│  · 没有 AI 你最大的困难是什么（脱离 AI 可完成度）          │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│          Part 2: 场景分流题（1 题）                       │
│                                                         │
│  "在以下任务中，你日常使用 AI 最多的是哪类？"              │
│   A. 写作与汇报（写报告、总结、邮件等）                    │
│   B. 学习与资料整理（读文章、做笔记、复习等）              │
│   C. 基础编程（写代码、改 bug、做小功能等）                │
│   D. 基础数据处理（表格分析、统计、做报表等）              │
│                                                         │
│  注：此题不参与评分，仅用于分流后续题目                    │
└──────────────────────┬──────────────────────────────────┘
                       ↓
           ┌───────────┼───────────┐
           ↓           ↓           ↓
┌─────────────────────────────────────────────────────────┐
│          Part 3: 场景专属题（4-5 题）                     │
│                                                         │
│  根据分流结果，仅出现对应场景的题目：                      │
│  · 该场景的核心能力点检测                                 │
│  · 该场景下 AI 的具体介入位置                             │
│  · 该场景下脱离 AI 后的具体影响                           │
│  · 该场景下的情境判断题                                   │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│               提交 → 评分管线 → 结果页                    │
│                                                         │
│  · sceneId = 用户在分流题中选择的场景                     │
│  · 通用题 + 场景题 → 合并后一起送入评分管线               │
│  · 场景权重矩阵正常生效                                  │
│  · 结果仍然是单场景三档结论                               │
└─────────────────────────────────────────────────────────┘
```

#### 阶段二（可选）：多场景串联

如果用户精力允许，可在结果页提供"继续评估其他场景"入口，复用已答的通用题，只需再答场景专属题（减少重复感）。

### 2.6 通用题设计建议

以下是 4 道推荐的跨场景通用题（每道 3 个选项，从低风险到高风险）：

#### 通用题 1：起始方式（跨场景）
> **当你开始一项工作任务时，你通常怎么做？**
> - A. 先自己想清楚要做什么、怎么做，再决定是否用 AI
> - B. 大概想一下方向，然后让 AI 帮我拓展思路或给个框架
> - C. 直接把任务描述发给 AI，让它先给一版方案或产出

#### 通用题 2：AI 介入位置（跨场景）
> **在你的日常任务中，AI 通常在什么阶段介入？**
> - A. 在我完成核心部分之后，帮我检查、润色或优化
> - B. 在过程中帮我补充关键内容或提供参考
> - C. 从任务一开始就全程参与，我主要做审核和修改

#### 通用题 3：脱离 AI 可完成度（跨场景）
> **如果现在完全不能用 AI，你能把手上的工作做到可交付的水平吗？**
> - A. 没问题，最多慢一些
> - B. 会比较吃力，某些环节可能质量下降
> - C. 很困难，我已经不太习惯完全不靠 AI 工作了

#### 通用题 4：依赖惯性（跨场景）
> **过去半年里，你觉得自己在没有 AI 辅助时的工作能力？**
> - A. 保持不变或有所提升
> - B. 某些方面有所下降，但还可以
> - C. 明显退步，离开 AI 后经常不知道从何开始

## 三、关键建议与注意事项

> [!WARNING]
> ### 必须守住的产品底线
> 1. **分流题不能省略**。没有场景锚点的判断是空的——这是产品存在的根基
> 2. **通用题不能评"AI 使用水平"**。通用题仍然必须围绕"你有没有把关键能力交给 AI"设计
> 3. **结果仍然是单场景结论**。不要因为有通用题就输出"跨场景综合画像"——那会违反产品第一原则
> 4. **保留原有的"直接选场景进入"路径**。有些用户明确知道自己要评估什么场景，不要强制他们走统一流程

> [!TIP]
> ### 实施建议
> 1. **先做数据层**（通用题设计 + seed 修改），这是基础
> 2. **再做 API 层**（接口 + 评分逻辑），这是核心
> 3. **最后做前端**（UI 改造），这是展示层
> 4. **保留双入口**：统一测评（MBTI 式）+ 直接选场景测评（原模式），让用户自选
